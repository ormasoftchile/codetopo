## Work Summary

Performance analysis of persist subsystem degradation at scale. Previous profiles showed per-file persist cost growing from 1.5ms (4K files) to 9.1ms (105K files) — a 6x degradation despite 0.0s lock contention, indicating non-threading scaling issues.

### Analysis Scope

- **Examined:** src/db/persister.h, src/cli/cmd_index.cpp, src/db/database.h, src/db/database.cpp
- **Profile data:** FSM run (4,145 files) vs Full run (105,694 files), same machine, cold-index both
- **Focus:** Identify root causes explaining the 7.6ms/file persist cost delta

### Findings Summary

**5 Root Causes Identified (80-100% of observed degradation):**

| RC | Issue | Est. Impact | Priority |
|---|---|---|---|
| RC-1 | `--turbo` flag is dead code — `enable_turbo()` never called; PRAGMAs silent fail | 2.8-3.8ms/file | **CRITICAL** |
| RC-2 | `nodes.stable_key` UNIQUE constraint creates undropable B-tree index | 1.2-1.7ms/file | HIGH |
| RC-3 | `mmap_size=256MB` undersized for 2GB databases at 100K scale | 1.5-2.0ms/file | HIGH |
| RC-4 | Foreign key checks during bulk insert (142 per file × B-tree probes) | 0.4-0.9ms/file | MEDIUM |
| RC-5 | `files.path` UNIQUE constraint duplicate-check overhead | 0.2-0.4ms/file | LOW-MEDIUM |

**Combined model:** 6.8-9.5ms/file (observed: 9.1ms) — 80-100% coverage.

## Key Insight

The dead `--turbo` flag is the critical blocker. `Connection::enable_turbo()` exists in connection.h:69 but is never called. At 100K files (211 COMMIT operations), the default `synchronous=NORMAL` + `wal_autocheckpoint=1000` causes:
- WAL fsync on every COMMIT (~1-5ms each on SSD)
- Auto-checkpoint triggers when WAL exceeds 1000 pages (random I/O on large DB files)

At 4K files (~8 commits): negligible. At 100K: dominant bottleneck.

## Recommendations Prioritized

**R1 (P0):** Wire up `--turbo` PRAGMAs in cmd_index.cpp → 9.1ms to 5-6ms/file (-35%)  
**R2 (P1):** Disable FK checks during bulk insert → 4.5-5.5ms/file (-45% cumulative)  
**R3 (P1):** Increase mmap_size to 2GB for bulk mode → 3.5-4.5ms/file (-55% cumulative)  
**R5 (P2):** Scale batch_size to 5000 for 100K+ workloads → 3.2-4.2ms/file (-58% cumulative)  
**R4 (P2):** Increase page_size to 8192 for fresh databases → 2.8-3.8ms/file (-62% cumulative)  
**R6 (P3):** Drop `nodes.stable_key` UNIQUE during bulk insert (advanced) → 2.0-3.0ms/file (-70% cumulative)

**Cumulative impact of P0+P1 fixes:** 9.1ms → 3.5-4.5ms/file (2-2.5x improvement). Full 100K run: ~915s to ~370-475s for persist alone (440-545s savings). Estimated total wall time reduction from 1474s to ~930-1030s (~35% faster). Persist drops from 62% to ~30-35% of wall time.
