\section{Related Work}
\label{sec:related-work}

\subsection{Tag-based tools}

\textbf{Universal Ctags} (and its predecessor \texttt{ctags}) produces a flat
tag file by applying regex patterns to source text. It supports over 150
languages via \texttt{optlib} extensions and is the de facto standard for
``jump to definition'' in lightweight editors. However, it records no call
graph, no import/include graph, and no FTS index. The tag file is editor-local
and has no query API. Research on call graph construction
(\cite{ryder1979callgraph}) established that a flat tag file is insufficient for
understanding program control flow.

\textbf{cscope} extends this approach for C-family code by building an
inverted reference table: ``who calls what'', ``called by'', text grep, and
file includes. Its coverage is effectively C-only, it has no semantic graph, no
FTS, and its interface is a curses TUI or a custom binary protocol with no
JSON/REST surface.

\subsection{Visual graph explorers}

\textbf{Sourcetrail} (archived 2021) used libclang (C/C++), Eclipse JDT (Java),
and a custom Python extractor to build a cross-reference graph stored in
SQLite --- conceptually the closest prior art to codetopo. It covered only four
languages, required compilation setup for C/C++, and had no query API (visual
explorer only). It is no longer maintained.

\textbf{Kythe}~\cite{google2014kythe} is Google's production system for
cross-language semantic cross-references. It uses build-system integration
(primarily Bazel) to hook into compilers, producing a fact/edge graph stored in
LevelDB. Kythe achieves full type resolution and supports C++, Go, Java, Python,
and TypeScript. Its principal limitation for local developer use is complexity:
it requires a Bazel build and a serving infrastructure. It has no MCP surface,
no FTS, and no incremental content hashing. codetopo trades type-resolution
completeness for zero-setup local deployment.

\subsection{Language Server Protocol servers}

LSP-based tools~\cite{microsoft2016lsp} such as \textbf{rust-analyzer} (Rust)
and \textbf{clangd} (C/C++) achieve deep semantic analysis through full
compiler integration. rust-analyzer uses an incremental computation graph
(Salsa) with type inference across generics. clangd uses the Clang AST with a
background disk index. Both are single-language and communicate over the LSP
protocol designed for editor interactions, not autonomous agent queries. Their
indexes are ephemeral (in-memory for rust-analyzer; session-local for clangd)
and not designed for batch or cross-file structural queries outside the LSP
session.

\subsection{AI-native tools}

\textbf{GitHub Copilot Code Search} uses the ``Blackbird'' engine: a trigram
plus symbol index (Rust, proprietary) combined with dense embedding retrieval.
It operates over all of GitHub and supports natural language and regex queries.
The index is remote, non-exportable, and has no MCP server surface exposing
the structural graph. Copilot is an MCP \emph{client}, not a server.

\textbf{Semgrep} performs AST-based structural pattern matching (e.g.,
\texttt{\$X.foo(\$Y)}) for security linting. Its Pro tier adds taint and
dataflow analysis. Semgrep has no persistent symbol graph, no call-graph
navigation, and no blast-radius analysis. It is purpose-built for security
pattern detection, not code navigation.

\textbf{codebase-memory-mcp}~\cite{deusdata2026codebasememory} indexes a
repository into a persistent knowledge graph using tree-sitter across 158
languages, with an additional ``Hybrid LSP'' layer for type-aware resolution in
nine language families. It stores the graph locally in SQLite-backed caches and
exposes 14 MCP tools plus a CLI for structural search, semantic retrieval, call
tracing, and architecture queries. It is open source (MIT), but its public
surface is centered on tool-owned graph queries rather than the smaller,
explicitly local SQLite schema and stable query surface that codetopo
prioritizes.

\textbf{CodeCompass}~\cite{ericsson2016codecompass} is an earlier code
comprehension framework for large projects, especially C/C++ and Java, with
plugin-based deep parsing and optional incremental re-parsing from compilation
databases. It persists parse results in SQLite or PostgreSQL and serves them
through a web UI, a web server, and a limited C/C++ LSP-style HTTP endpoint.
CodeCompass is open source (GPL-3.0), but it requires heavier setup than
codetopo, is build-configuration-dependent for strong results, and has no MCP
surface for autonomous agents.

\textbf{code-review-graph}~\cite{tirth2026codereviewgraph} builds a local
tree-sitter graph of files, symbols, calls, imports, tests, flows, and
communities, stores it in SQLite with FTS5, and optionally augments search with
embeddings. Its primary query surfaces are a CLI, 30 MCP tools, and GitHub
Action / editor integrations aimed at token-efficient review workflows. It is
open source (MIT) and close in spirit to codetopo, but it is deliberately
review-centric, and its own evaluation documents weaker flow detection and
conservative impact precision outside that review setting.

\textbf{CodeGraphContext}~\cite{codegraphcontext2025cgc} turns repositories
into graph databases using tree-sitter, with optional SCIP indexers for more
precise C/C++ and C\# resolution. It offers both a CLI and an MCP server with
roughly two dozen tools, backed by FalkorDB Lite, KuzuDB, LadybugDB, Neo4j, or
Nornic rather than a single embedded format. CodeGraphContext is open source
(MIT), but its storage/backend story and even its published counts (languages
and tools) vary across documents, making it less lightweight and less
internally uniform than codetopo's single-file SQLite deployment.

\textbf{Repowise}~\cite{repowise2026repowise} extends the ``AI-native'' idea
beyond code structure: tree-sitter-derived graph indexing is combined with git
analytics, generated documentation, architectural-decision mining, and
deterministic code-health scoring. It exposes 10 MCP tools in single-repo mode
(13 in workspace mode), a CLI, and a local dashboard, storing data in
\texttt{.repowise/wiki.db} by default (SQLite; PostgreSQL optional). Repowise
is open source (AGPL-3.0), but its initialization is substantially heavier than
codetopo's and many of its headline features assume a broader documentation /
LLM pipeline rather than a minimal local structural index.

\subsection{Gap analysis}

\subsubsection{What codetopo uniquely provides}

\begin{description}
  \item[Unified MCP server with 27 graph-query tools.] No other tool in the
        landscape exposes a persistent, queryable call graph + FTS + file-
        dependency graph as an MCP server that AI agents can call directly.
  \item[Polyglot call graph in a local SQLite database.] Sourcetrail also used
        SQLite and cross-references but covered only four languages and is
        abandoned. codetopo covers 13 languages with a unified schema, persists
        between sessions, and is queryable via CLI.
  \item[Blast-radius / impact analysis.] Transitive dependency queries
        (\texttt{impact\_of}, \texttt{shortest\_path}) are not offered by any
        LSP server. Kythe could support it in theory but offers no out-of-box
        query.
  \item[Incremental re-indexing via content hashing.] Combined with a file
        watcher, codetopo re-indexes only changed files in real time. clangd
        does incremental parsing but only within one LSP session; rust-analyzer
        discards state on restart.
  \item[One-command editor auto-configuration.] \texttt{codetopo init} writes
        MCP server configuration into VS Code, Cursor, Windsurf, and Claude
        Desktop automatically.
\end{description}

\subsubsection{Known gaps vs.\ competitors}

\begin{description}
  \item[No type resolution.] codetopo's call edges are AST-derived
        approximations (\texttt{callers\_approx}, \texttt{callees\_approx}).
        rust-analyzer and clangd resolve overloads, generics, and templates
        with full type inference.

        \emph{Tradeoff:} Adding precise resolution would force codetopo beyond a
        uniform tree-sitter pipeline into language-specific semantic front-ends
        or imported SCIP/LSIF data, increasing setup friction and making the
        current single-schema abstraction harder to preserve across languages.
        \emph{Cost:} XL (multiple person-months), with the hardest part being a
        durable cross-language representation for resolved types, overloads, and
        external dependencies that still fits incremental local indexing.
        \emph{Value:} It would materially improve ``go to real callee'',
        polymorphic impact analysis, and agent workflows that must distinguish
        same-named methods, constructors, or generic instantiations.
  \item[No taint/dataflow analysis.] Security and refactoring use cases
        benefit from tracking whether data flows between two symbols. Semgrep
        Pro and Kythe provide this; codetopo does not.

        \emph{Tradeoff:} Dataflow would add a second analysis stack beyond the
        current symbol/reference graph, with substantially higher correctness
        risk from path sensitivity, aliasing, interprocedural summaries, and
        language-specific semantics. \emph{Cost:} XL, because tree-sitter alone
        is insufficient; the hardest part is introducing and maintaining an IR
        or SSA-like graph that can represent control/data dependencies across
        supported languages without turning the indexer into a compiler.
        \emph{Value:} It would unlock source-to-sink security queries, more
        credible ``can this input reach here?'' agent answers, and safer
        automated refactoring for parameter, return-value, and field propagation.
  \item[No formal LSP endpoint.] codetopo speaks MCP. Adding an LSP adapter
        would expand editor coverage beyond the VS Code family.

        \emph{Tradeoff:} An LSP surface would widen compatibility, but it would
        also impose JSON-RPC session management, text-document lifecycle
        handling, and editor-oriented semantics that are distinct from the
        current tool-style MCP contract. \emph{Cost:} M--L, with the hardest
        part being a robust server loop plus enough workspace/\allowbreak
        \texttt{textDocument} methods to be useful without duplicating a full
        language server for each language. \emph{Value:} It would enable
        lightweight editor integrations, make codetopo usable in clients that
        cannot host MCP, and let developers reuse the local SQLite index for
        jump-to-definition, references, and call-hierarchy views.
  \item[No remote/shared index.] The index is per-developer local. Kythe and
        GitHub Code Search support team-shared indexes; codetopo does not.

        \emph{Tradeoff:} A shared index would sacrifice some of codetopo's
        simplicity and privacy by introducing synchronization, trust, and
        freshness problems that do not exist for a single local database.
        \emph{Cost:} L--XL, because the hardest part is not storage but stable
        cross-machine identity: syncing nodes/edges, reconciling divergent
        repository states, and handling deletes or rebases without corrupting
        graph consistency. \emph{Value:} It would help onboarding, CI-produced
        indexes for very large repositories, and agent workflows that need a
        common team view rather than per-laptop cold starts.
  \item[No symbol-importance ranking.] FTS5 BM25 scores text relevance but
        codetopo has no PageRank-style scoring on the call graph.

        \emph{Tradeoff:} Ranking would improve prioritization, but it also
        introduces another tunable heuristic whose outputs can appear
        authoritative even when they reflect graph incompleteness or language
        biases in the approximate call edges. \emph{Cost:} S--M, since the core
        computation is comparatively small; the hardest part is choosing a
        ranking formulation and update strategy that remains stable and
        interpretable under incremental re-indexing. \emph{Value:} It would
        improve search result ordering, help agents surface ``central'' symbols
        first, and support architecture, hotspot, and review workflows that need
        to separate core APIs from peripheral helpers.
\end{description}
